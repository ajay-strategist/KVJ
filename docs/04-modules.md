# 04 — Module Documentation

FlowDesk is organised into the functional modules below. Each maps to a route file + controller(s) on the backend and one or more pages on the frontend. Modules can be individually enabled/disabled via the `Settings.enabledModules` toggles (attendance, leaves, timesheets, projects, chat, teams).

---

## 1. Authentication & Accounts

**Backend:** `authController`, `authRoutes`, `authMiddleware`. **Frontend:** `Login.jsx`, `Signup.jsx`.

- **Register** (`POST /api/auth/register`): creates a user with `status: 'Pending'` — no self-service activation; an admin must approve.
- **Login** (`POST /api/auth/login`): rejects `Pending` and `Deactivated` accounts, verifies the bcrypt hash, issues a 30-day JWT, sets it as an httpOnly cookie **and** returns it in the JSON body (the SPA also stores it in `localStorage`).
- **Logout** (`POST /api/auth/logout`): clears the cookie.
- **Roles:** `Admin`, `Manager`, `Employee`; plus a boolean `isTrainer` capability flag and a `grade` (Intern/Junior/Senior/Lead) and `salaryRate` used for cost calculations.
- A default admin can be seeded via `seedAdmin.js` (`admin@flowdesk.com` / `admin123`).

## 2. User Administration

**Backend:** `userController`, `adminTrainerController`, `profileController`, `performanceReportController`, `userRoutes`. **Frontend:** `AdminUsers.jsx`.

- Admin lists all users; Managers see only their own team (filtered in the controller).
- Approve / reject pending registrations (approval triggers a Gmail approval email **and** seeds that year's leave balances).
- Update role, status, team, grade, salary rate; deactivate (which force-logs-out the user over Socket.IO via `forceLogout`).
- Toggle `isTrainer` (`PATCH /api/admin/users/:id/trainer`).
- **Member profile dashboard** (`GET /api/users/:id/profile`) and **performance report** (JSON + XLSX) aggregating attendance, leaves, expenses and trainer logs over a date range.
- CSV/XLSX exports of users and performance.

## 3. Teams

**Backend:** `teamController`, `teamRoutes`. **Frontend:** `Teams.jsx`.

- Teams have a name, description and a single `manager` (User ref).
- Admin-only create/update/delete; any authenticated user can read teams and team members.
- Users reference their team via `User.team`; changing a user's team emits `userUpdated` so the client re-joins the correct team socket room.

## 4. Projects

**Backend:** `projectController`, `projectRoutes`. **Frontend:** `Projects.jsx`, `ProjectManagement.jsx`.

- Projects have name, description, `client` (**free-text string**, not linked to the Client collection), start/end dates, status (Active/On Hold/Completed), a `projectGroup` label, and arrays of `managers` and `members`.
- Admin creates/deletes; Admin+Manager update. All authenticated users can view.
- Endpoints to fetch a project's members and its tasks; CSV export (admin).
- Live progress broadcasts to `project_<id>` socket rooms.

## 5. Tasks (Work Management)

**Backend:** `taskController` (725 LOC — the largest controller), `taskRoutes`, plus `timerController`/`timerRoutes` and `taskStartAccess` util. **Frontend:** primarily inside `Projects.jsx`.

The richest module. Capabilities:

- **CRUD** with project/team/assignee association. A `null` assignee means the task sits in a **team pool** that members can **claim** (`PUT /api/tasks/:id/claim`), broadcast as `poolTaskClaimed`.
- **Configurable statuses/priorities** driven by `Settings.taskStatuses` / `taskPriorities`.
- **Overdue flagging** — reactively in `getTasks` and proactively via a nightly cron.
- **Recurring tasks** (none/daily/weekly/monthly) regenerated by a cron job.
- **Time logging** (`timeLogs[]` on the task) and a **real-time timer** (start/pause/end) with accumulated seconds. Non-admins must be clocked-in and off-break to start a task or timer (`taskStartAccess`).
- **Task comments** (`TaskComment` collection) with socket toast notifications.
- **Task transfer** workflow: employee initiates transfer → recipient accepts/rejects → **admin approves/rejects** (`TaskTransfer` collection, dual `status` + `adminStatus`).
- **Manager→Admin assignment approval:** a manager's task assignment can require admin approval (`managerApprovalPending`, `pendingAssignee`).
- **Archiving:** completed tasks archived; a cron deletes archived tasks older than one year.
- CSV export (admin).

## 6. Attendance (Time Clock)

**Backend:** `attendanceController` (409 LOC), `attendanceRoutes`, `autoClockOut` cron. **Frontend:** `Attendance.jsx` (personal), `AdminAttendance.jsx` (admin), `TeamStatusPanel.jsx`.

- **Clock-in/out** with optional **geolocation** capture (lat/long/accuracy/city) and **break in/out** tracking; `totalHours` computed net of breaks.
- **Labels:** Normal, Weekend, Public Holiday, Auto-closed, Half Day (Morning/Evening Leave).
- **Auto clock-out** cron at 23:59 closes any dangling open sessions and labels them `Auto-closed`.
- **Team status** (`GET /api/attendance/team-status`, Admin/Manager) and **all attendance** views.
- **Admin correction** (`PUT /api/attendance/:id`) writes an immutable `correctionLog[]` audit trail (who/when/note/previous/updated).
- **Heartbeat** endpoint + 60s client loop maintain `lastActiveAt` for Active/Away presence.

## 7. Leave Management

**Backend:** `leaveController` (450 LOC), `leaveRoutes`, `leaveBalanceReset` + `overdueReports` crons. **Frontend:** `Leaves.jsx`.

- Leave requests with from/to dates, reason, nature (Medical/Personal), half-day support (Morning/Evening slot), and computed `daysTaken`.
- **Approval** by Admin/Manager with `managerComment`; approval sets `approvedBy`.
- **Per-user leave balances** (`LeaveBalance` per `LeaveType` per year) with a `remainingDays` virtual; seeded on user approval and reset annually by cron each Jan 1.
- **Leave types** (`LeaveType`) are configurable (days-per-year, colour, custom flag).
- **Medical certificate workflow:** medical leaves get a `reportDeadline` and `reportStatus` (Not Required/Pending/Submitted/Overdue); a nightly cron marks overdue reports and notifies the employee via DM + a General-channel post.
- **Public holidays** CRUD (admin) influence attendance labelling.
- CSV exports for leaves and attendance.

## 8. Timesheets / Worklogs

**Backend:** `timesheetController`, `timesheetRoutes` (mounted at both `/api/timesheets` and `/api/worklogs`). **Frontend:** `Timesheets.jsx`.

- Log hours against a task/project with date, notes and a computed `cost` (hours × user `salaryRate`).
- **Auto-approved on creation** (the route file explicitly notes there is no approval step, despite a `status` enum existing on the schema).
- Owner or elevated roles can update/delete; Admin/Manager can view team timesheets; admin XLSX export.

## 9. Expenses

**Backend:** `expenseController` (432 LOC), `expenseRoutes`, `expenseTypeController`. **Frontend:** `Expenses.jsx` (1,344 LOC).

- Expense claims with category (Training/Office), structured fields (location, batch, course, expense type, self-travel vehicle/distance), amount, number of persons, notes, and a **bill/receipt image** uploaded to **Google Drive** (filename `{user}-{amount}-{type}-{date}.ext`).
- **Approval workflow:** Pending → Approved → Paid (or Rejected), with `managerNotes`.
- **Bulk approve / bulk pay** (admin) and admin CSV export.
- `ExpenseType` is a configurable reference collection (name + Training/Office category).
- Legacy fields (title, generic category, currency, `receiptUrl`, `dateIncurred`) retained for backward compatibility alongside the newer structured fields.

## 10. Chat (Team Messaging)

**Backend:** `chatController` (342 LOC), `chatRoutes`, Socket.IO. **Frontend:** `Chat.jsx` (1,107 LOC).

- **Channels** (General / Team / Custom / DM). Admin/Manager create channels and manage membership.
- **Direct messages** via get-or-create DM channels.
- **Threaded replies** (`threadId`) and **@mentions** (`mentions[]`) with unread mention counts.
- **File attachments** (multer; currently memory storage as Cloudinary is stubbed).
- Real-time delivery via channel rooms; global toasts in `DashboardLayout` for new messages, mentions and thread replies.

## 11. Trainer Logging & Training Batches

**Backend:** `trainerLogController`, `trainerLogRoutes` (+ `isTrainer` gate); `trainingBatchController`, `trainingBatchRoutes`. **Frontend:** `TrainerLog.jsx`, `ManageBatches.jsx`.

- **Trainer logs** (trainer-only): date, organisation, type (Class/Work/Meeting/Supervision/Marketing), mode (Online/Offline), start/end time, computed duration. Admin can view all and export.
- **Training batches**: `College` + `Course` reference collections and a `TrainingBatch` (unique per college+course+batch) used to categorise trainer/expense activity.

## 12. Dashboards & Reporting

**Backend:** `dashboardController`, `performanceReportController`, plus per-module CSV/XLSX exports. **Frontend:** `Dashboard.jsx`, `AdminDashboard.jsx`.

- **Admin dashboard stats** (`GET /api/dashboard/admin`): active user count, project totals/completed, today's attendance count, pending leaves & expenses (with pending-expense total).
- **Performance report** per user aggregating attendance, approved leaves, expenses, trainer logs and holidays over a date range (JSON + XLSX).
- Nearly every module offers a CSV/XLSX export (users, tasks, projects, timesheets, expenses, leaves, attendance, trainer logs).

## 13. Settings & Configuration

**Backend:** `settingsController`, `settingsRoutes`. **Frontend:** surfaced within admin screens.

- Singleton `Settings` document: configurable task statuses & priorities, per-module enable/disable toggles, org name and logo.
- Read by any authenticated user (the client needs the label config); written by Admin only.

## 14. Calendar Integration

**Backend:** `calendarController`, `calendarRoutes`, `calendarService`. **Frontend:** integrated into leave/task views.

- Per-user **Google Calendar OAuth**: auth-URL generation, connect (token exchange), disconnect, status. Tokens stored on `User.googleCalendar`.
